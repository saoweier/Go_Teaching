# 产品规划

## 1. 产品定位

这是一个围棋教学方向的 Web 软件，不追求“像普通对弈平台一样能下棋”这么简单，而是要在下棋过程中持续回答三个问题：

- 这一步为什么好或不好？
- 接下来大概率会往什么方向发展？
- 如果我现在看不清，系统能不能帮我把后续几步讲明白？

因此产品核心不是单纯 AI 对弈，而是「对弈 + 分析 + 教学」三位一体。

## 2. 核心模式设计

### 2.1 全盘对局模式

用户选择执黑或执白，在 19x19 标准棋盘上与 AI 对弈。

需要提供：

- 正常落子、提子、停一手、认输、重新开始。
- 实时显示当前胜率、目差趋势、推荐点。
- 支持用户在任意时刻请求“分析当前盘面”。
- 支持查看 AI 推荐的 1 到 3 条主要变化分支。

### 2.2 局部攻防教学模式

你提到的“1/4 小棋盘”非常适合做局部训练，但建议产品上定义为“局部聚焦视图”，而不是硬编码成固定四分之一棋盘。

建议做法：

- 在全盘任意区域框选一个局部区域，自动生成局部棋盘。
- 默认提供 9x9、11x11、13x13 三种局部训练尺寸。
- 允许从预置题目进入，也允许从当前对局一键切换到局部攻防视图。
- 局部视图与全盘状态共享同一个真实盘面，不产生规则冲突。

这样比固定 1/4 棋盘更灵活，后续也更适合做死活、定式后续、打劫局部训练。

### 2.3 教学分析模式

这是产品差异化重点。

用户点击分析后，系统不只给一个胜率，而是输出：

- 当前局面判断：优劣、形势、紧急程度。
- 候选点排序：推荐点、可下点、明显问题手。
- 后续趋势：如果走 A，通常会形成什么结构；如果走 B，风险是什么。
- 教学解释：用适合初学者的语言说明“原因”，而不是只说“胜率下降 7%”。

## 3. 关键用户流程

### 3.1 初学者边下边学

1. 用户选择先后手和难度。
2. 用户在大棋盘落子。
3. AI 回应一步。
4. 用户点击“分析这一步”。
5. 系统展示：
   - 这一步是否合理。
   - AI 更推荐的点。
   - 未来 5 到 15 手的主要变化趋势。
   - 一段简洁教学解释。
6. 用户继续下，或者切到局部视图继续学习。

### 3.2 局部攻防训练

1. 用户从题库或全盘局面进入局部训练。
2. 系统聚焦局部区域。
3. 用户尝试下关键一手。
4. AI 应对。
5. 系统展示局部胜负关键、眼位、联络/分断、是否成活。

### 3.3 终局复盘

对你的设想，我建议终局不要只做“死活教学”或“数目”二选一，而是做一个终局总结面板，包含三层：

- 死活复核：哪些块安全，哪些块未净活，哪里仍有官子或劫争残留。
- 数目判断：地、贴目、胜负估计。
- 教学总结：中盘关键转折、最大损失手、最值得复盘的 3 个节点。

这样比单独做数目更有教学价值，也更符合初学者需求。

## 4. 功能优先级

## P0：必须先做

- 19x19 棋盘对弈。
- 基本规则引擎：禁入点、提子、停一手、终局。
- 接入 KataGo 进行局面分析。
- 展示胜率、目差、推荐点、主变化。
- 分析面板支持“当前一步为什么不好/好”。
- 局部聚焦视图，可从全盘切换进入。

## P1：明显提升体验

- 题库系统：死活题、手筋题、局部攻防题。
- 变化图回放和逐步讲解。
- 教学解释模板引擎。
- AI 难度分级：偏教学、偏实战、偏让子练习。
- 终局总结报告。

## P2：后续增强

- 接入本地或云端 LLM，把分析结构化结果转成自然语言。
- 教师模式：自定义题目、设置正确分支和错误分支。
- 对局记录、复盘历史、用户成长档案。
- 语音讲解或课堂投屏模式。

## 5. 关于模型方案的建议

## 5.1 不建议只用通用小模型算棋

你已经验证过，本地通用小模型在围棋计算和局面理解上不稳定。这是预期内的，因为围棋分析需要：

- 严格规则判断。
- 大量变化搜索。
- 目差和胜率估计。
- 对局部与全局价值的稳定比较。

这些都不是通用小模型擅长的。

## 5.2 推荐主方案

建议采用双层架构：

- 棋力层：KataGo 或同类专业围棋引擎，负责搜索、胜率、候选点、变化图、终局估计。
- 讲解层：规则化解释模板 + 可选 LLM，负责把引擎结果变成适合初学者的教学说明。

## 5.3 第一版是否必须接 LLM

不必须。

第一版完全可以不接 LLM，先做高质量结构化解释：

- 这一步问题在于薄味太重。
- 这一步导致角部三子被分断。
- 这一步虽然是先手，但本地价值不如补强大龙。
- 推荐先走 X，因为它同时处理了出头和联络。

只要解释模板设计好，初期教学价值已经很高，而且结果更可控。

## 6. 我建议的产品原则

- 规则和分析正确性优先于语言华丽度。
- 局部训练必须和全盘状态统一，避免“局部能走、全盘不成立”的割裂。
- 任何分析结果都要可回放，不能只给结论。
- 面向初学者时，默认少讲术语，多讲后果。
- 不把“AI 推荐手”当答案，而是强调“为什么这手更符合当前目标”。

## 7. MVP 范围建议

为了尽快做出可用版本，建议 MVP 控制在下面这条链路：

1. 用户进入网页。
2. 选择执黑/执白。
3. 与 AI 在大棋盘对局。
4. 随时点击分析。
5. 查看推荐点、主变化、简短解释。
6. 任意时刻切换到局部聚焦视图。
7. 对局结束后查看终局总结。

只要这条链路跑通，产品已经有明显教学价值。

## 8. 后续落地建议

如果你认可这个方向，下一步最值得做的不是继续泛泛讨论，而是直接产出三样东西：

1. 信息架构和页面草图。
2. 后端模块拆分和 API 清单。
3. MVP 开发排期。

这三样确定后，就可以开始真正建项目。
