# 技术架构

## 1. 总体架构建议

建议采用标准 B/S 架构，但后端内部拆成三个核心服务层：

- Web 应用层：负责页面、棋盘交互、用户会话、复盘展示。
- 围棋分析层：负责规则校验、局面管理、调用 KataGo。
- 教学解释层：负责把分析结果转成结构化教学内容，可先模板化，后接 LLM。

推荐形态：

- 前端：Vue 3 或 React。
- 后端 API：FastAPI 或 Node.js/NestJS。
- 分析引擎：KataGo + GTP/分析接口适配服务。
- 数据存储：PostgreSQL。
- 缓存/任务：Redis。
- 对局与题库文件：SGF 存储。

如果让我替你选第一版技术栈，我会建议：

- 前端：Vue 3 + TypeScript + Vite。
- 后端：FastAPI + Python。
- 原因：Python 更容易和 KataGo、SGF、围棋相关工具链集成。

## 2. 模块拆分

## 2.1 前端模块

### 棋盘模块

- 19x19 全盘渲染。
- 局部区域渲染。
- 落子、提子、标记、推荐点高亮。
- 变化图回放。

建议从一开始就把棋盘做成“统一渲染组件 + 不同视口配置”，不要做两套棋盘组件。

### 分析面板

负责展示：

- 当前胜率和目差。
- 候选点列表。
- 主变化分支。
- 教学解释文本。
- 本手评价标签，例如“好手”“疑问手”“缓手”“恶手”。

### 复盘面板

- 历史着手列表。
- 跳转到关键节点。
- 对局结束总结。

## 2.2 后端模块

### Game Service

负责：

- 棋局状态维护。
- 合法性判断。
- 落子、提子、停一手。
- SGF 导入导出。
- 局部视图与全盘映射。

### Analysis Service

负责：

- 调用 KataGo 分析当前盘面。
- 获取候选点、policy、winrate、scoreLead、ownership。
- 生成主变化和变化树。
- 对接异步分析任务。

### Teaching Service

负责：

- 把分析结果转成教学标签。
- 产出“为什么这步不好/好”的结构化原因。
- 组织终局总结。
- 可选接入 LLM 做语言润色和分层解释。

### Content Service

负责：

- 题库管理。
- 定式、死活、局部攻防素材。
- 用户训练记录。

## 3. 核心数据流

### 3.1 对局流程

1. 前端提交用户落子。
2. Game Service 校验合法性并更新盘面。
3. Analysis Service 请求 AI 落子或盘面分析。
4. 返回 AI 落子和主变化。
5. Teaching Service 根据结果生成解释。
6. 前端展示盘面和教学面板。

### 3.2 局部聚焦流程

1. 用户在全盘框选区域或进入预置题。
2. Game Service 根据坐标裁剪出局部视图。
3. Analysis Service 对局部重点进行分析，但引用的是同一真实盘面。
4. 前端展示局部区域，并允许回到全盘。

这里要注意一个关键点：

局部视图可以是“显示范围变小”，但分析时不能完全脱离全盘上下文，否则会出现局部看起来成立、全局却不成立的问题。实践上应支持两种分析：

- 全局上下文分析：主流程默认使用。
- 局部强化分析：只在教学题或死活题中作为辅助使用。

## 4. KataGo 接入建议

## 4.1 为什么选 KataGo

它更适合你这个场景，因为它不只是“下棋强”，还可以提供教学需要的数据：

- 候选点概率。
- 胜率变化。
- 目差估计。
- 归属图和厚薄判断基础数据。
- 变化分支。

## 4.2 接入方式

不建议前端直接调引擎。建议单独封装一个 `engine-adapter`。

职责：

- 管理 KataGo 进程生命周期。
- 统一分析请求格式。
- 控制并发和超时。
- 缓存相同盘面的分析结果。
- 把底层字段整理成前端友好的结构。

## 4.3 分析结果结构建议

后端可统一输出：

- `top_moves`
- `winrate`
- `score_lead`
- `ownership_map`
- `principal_variations`
- `move_quality`
- `teaching_points`

这样未来替换引擎或切换不同模型时，前端和教学层都不需要跟着大改。

## 5. LLM 接入建议

## 5.1 初期方案

先不上 LLM 直出结论，而是走：

- KataGo 原始数据
- 教学规则提取
- 模板化讲解

优点：

- 结果稳定。
- 可测试。
- 不会胡乱编棋。
- 开发成本更可控。

## 5.2 增强方案

后期再让 LLM 做两件事：

- 把结构化教学点翻译成自然语言。
- 根据用户水平做解释层级调整，例如初学者版、进阶版。

LLM 不参与：

- 合法性判断。
- 最佳着搜索。
- 胜率和目差计算。

## 5.3 本地 LM Studio 是否适合

适合做辅助讲解层，不适合做核心分析层。

如果你本地机器性能有限，建议把本地 LLM 设为可选插件，而不是系统必需组件。

## 6. 前后端 API 草案

## 6.1 对局

- `POST /api/games`
  - 创建棋局，参数包括棋盘尺寸、执黑方、AI 难度、模式。
- `POST /api/games/{id}/moves`
  - 用户落子。
- `POST /api/games/{id}/pass`
  - 停一手。
- `POST /api/games/{id}/resign`
  - 认输。
- `GET /api/games/{id}`
  - 获取当前棋局状态。

## 6.2 分析

- `POST /api/games/{id}/analyze`
  - 分析当前局面。
- `POST /api/games/{id}/analyze/move`
  - 专门分析刚走的一步。
- `POST /api/games/{id}/variations`
  - 请求更多变化分支。

## 6.3 局部教学

- `POST /api/games/{id}/focus`
  - 创建或更新局部聚焦区域。
- `GET /api/games/{id}/focus`
  - 读取当前局部区域。
- `POST /api/problems/{id}/attempt`
  - 提交局部题答案。

## 6.4 终局总结

- `POST /api/games/{id}/final-review`
  - 生成终局总结，包括死活复核和数目判断。

## 7. 实施阶段建议

## 阶段 1：MVP

- 完成基础棋盘和对局。
- 接入 KataGo。
- 支持分析当前局面和显示主变化。
- 支持局部聚焦视图。

交付结果应当是：已经能下、能看、能分析。

## 阶段 2：教学化

- 增加教学标签和解释模板。
- 增加关键手识别。
- 增加终局总结。

交付结果应当是：已经不只是引擎面板，而是具备“教”的能力。

## 阶段 3：内容化

- 增加题库系统。
- 增加用户记录和成长轨迹。
- 增加教师出题能力。

交付结果应当是：从工具升级为教学平台。

## 8. 风险与控制

### 风险 1：LLM 解释看起来像懂棋，其实不可靠

控制方式：

- 让 LLM 只解释结构化结论，不直接自由分析盘面。

### 风险 2：局部分析脱离全局导致误导

控制方式：

- 默认使用全局上下文分析。
- 局部题单独标注为“题目模式”。

### 风险 3：KataGo 分析延迟过高

控制方式：

- 做异步任务和结果缓存。
- 区分“快速分析”和“深度分析”两档。

### 风险 4：初学者看不懂专业术语

控制方式：

- 教学层输出分级解释。
- 首版减少术语，强调后果和目的。

## 9. 我给你的直接建议

如果目标是先做出一个真正能用的产品，最稳的路线不是：

- 先找一个本地小模型让它学会围棋。

而是：

1. 先把 KataGo 作为核心分析能力接进来。
2. 把前后端和对局流程打通。
3. 先做结构化教学解释。
4. 最后再决定是否要把 LM Studio 作为本地讲解增强层。

这条路线风险最低，也最容易做出第一个有说服力的版本。
